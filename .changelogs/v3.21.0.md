# Release v3.21.0

Released: 2026-07-01

## Highlights
- `--review-with` now accepts an **arbitrary GitHub reviewer** via the `@<login>` form — any GitHub user or App/bot (e.g. `--review-with @octocat`, `--review-with claude,@org-review-bot`, `--review-with @some-app[bot]`). slashdo requests that login's review on the PR, waits for it, and fixes whatever it surfaces — the same request → poll → fix → resolve flow the `copilot` reviewer already uses. In `--merge` mode the review must complete for the merge to proceed; a reviewer that never submits leaves the PR open rather than merging on an absent review. slashdo never posts an approval itself.
- **GitHub Enterprise support for `gh api` calls.** Commands that call `gh api` (the reviewer-poll loops, and `/do:rpr`, `/do:next`, `/do:review` PR mode, epic sub-issue resolution) now derive the API host from the repo's remote and pass it explicitly, instead of defaulting to github.com. On an Enterprise host the review loops previously hit the wrong host and timed out; they now target the correct host with no configuration. github.com repos are unaffected.

## Added
- **Arbitrary GitHub reviewers (`@<login>`)** — a new reviewer type accepted by every command that runs the multi-reviewer loop (`/do:pr`, `/do:release`, `/do:review`, `/do:better`, `/do:better-swift`, `/do:depfree`) and savable as a default via `/do:config`. It generalizes the Copilot loop to any GitHub login (a user, or an App whose login ends in `[bot]`):
  - New shared procedure `lib/github-reviewer-loop.md` requests the review (`gh api .../requested_reviewers`), polls GraphQL for a submitted review from that login, applies fixes, resolves threads, and pushes — bounded by `--review-iterations` and a generous wait suited to slower (human) reviewers.
  - A failed review request is non-fatal: slashdo keeps polling, so review Apps that post on their own (without being explicitly requested) still work.
  - Statuses: `clean`/`capped` (eligible to merge), `timeout`/`not-requestable`/`error`/`guardrail` (inconclusive — never merged).
  - GitHub only; on GitLab the `@<login>` (and `copilot`) passes are skipped with a warning and an `inconclusive` aggregate.
  - Login validation: `^[A-Za-z0-9][A-Za-z0-9-]*(\[bot\])?$`; the login (lowercased) is part of the `--review-with` dedup identity.
  - `/do:rpr` is intentionally not in scope: it already resolves review threads from any author (bot or human), but it runs a copilot-specific request/monitor flow rather than the multi-reviewer loop, so *requesting* an arbitrary `@<login>` reviewer there is a follow-up.

## Changed
- `/do:pr` now partitions reviewers into **local** (run before the PR, against the working tree) and **PR-side** (`copilot` + every `@<login>`, run after the PR is opened) — the section formerly titled "Run the Copilot Review" is now "Run the PR-side Reviews". The merge gate is unchanged: it is driven by the aggregate review status, which is reviewer-agnostic.

## Fixed
- **`gh api` no longer defaults to github.com on GitHub Enterprise repos.** `gh api` (both REST and GraphQL) ignores the git remote and targets github.com unless told otherwise — unlike `gh pr`/`gh repo`/`gh issue`, which resolve the host from the remote. On an Enterprise repo this made the reviewer-poll loops silently poll github.com and time out, `gh api user` return the wrong identity, and REST calls 404. slashdo now derives the host once from the `origin` remote (or the PR URL's host, in `/do:review` PR mode) and passes `--hostname` on every `gh api` call, with a per-host auth precheck that surfaces an actionable `gh auth login --hostname <host>` message instead of a mysterious timeout. New shared `lib/gh-host.md` documents the derivation. `/do:review` PR mode also host-qualifies `gh pr --repo`, and `/do:next`'s preflight now accepts Enterprise hosts (it previously hard-aborted on any non-github.com host).
